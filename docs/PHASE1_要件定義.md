## 0. 最重要原則（システム全体を貫く設計思想）

> **本システムは医師の診断を代替しない。**
> AIの役割は「問診の整理」「不足情報の追加質問」「緊急度の目安提示」「医師向け要約」「受診科目の候補提示」に限定する。
> **診断名の断定・治療方針の決定・処方の指示は、いかなる画面・いかなるAI出力においても行わない。**
> 最終判断は常に医師が行い、システムはそのための情報を整えるだけの「補助ツール」である。

この原則は、後述する法的位置づけ（医師法17条・薬機法のプログラム医療機器該当性）の根拠でもあり、
**マーケティング文言・UI文言・AIプロンプト・利用規約のすべてがこの原則と矛盾しないこと**を受け入れ条件とする。

## 1. プロジェクト概要

### 1.1 目的
患者がスマホから症状を入力すると、AIが不足情報を追加質問で補い、医師に渡す問診レポート（SOAP形式）を自動生成する。
これにより、

- 患者：受診前の不安整理・適切な受診行動（緊急度の目安・診療科の候補）
- 医師：問診時間の短縮・情報の抜け漏れ防止・診療前の事前把握
- クリニック：オンライン診療の予約〜診療〜記録の一気通貫の運用

を実現するマルチテナントSaaSを提供する。

### 1.2 対象ユーザー（テナント）
| セグメント | 主な利用シーン | 特記事項 |
|---|---|---|
| オンライン診療クリニック | 予約前問診・ビデオ診療 | コア顧客 |
| 内科 | 来院前Web問診 | 汎用問診テンプレート |
| 皮膚科 | 画像アップロード問診 | 画像の取り扱い要件が重い |
| 小児科 | 保護者による代理入力 | 年齢別質問分岐・体重確認 |
| メンタルクリニック | 心理系問診 | **自傷他害リスク検知が必須**（後述） |
| 訪問診療 | 家族・介護者による代理入力 | 代理入力者の記録 |
| 企業の健康相談窓口 | 産業保健・健康相談 | 診療ではなく「相談」導線（受診勧奨まで） |

### 1.3 アクター定義
| アクター | 説明 |
|---|---|
| 患者（patient） | 症状を入力する本人。代理入力（保護者・家族）を含む |
| 医師（doctor） | 問診レポートを確認し、診療・判断を行う |
| クリニック管理者（clinic_admin） | 医師・診療枠・テンプレートの管理 |
| システム管理者（super_admin） | SaaS運営側。テナント管理・監査 |
| AIエンジン | OpenAI / Anthropic API。**アクターだが決定権を持たない** |

## 2. システムの法的位置づけ（設計の前提）

### 2.1 プログラム医療機器（SaMD）非該当としての設計
薬機法上、「疾病の診断・治療・予防に使用されることが目的とされているプログラム」は医療機器に該当し、承認・認証が必要になる。
本システムは厚労省「プログラムの医療機器該当性に関するガイドライン」を前提に、**非該当（非医療機器）の範囲に収まるよう機能を設計する**。

非該当を維持するための設計上の縛り：

1. **診断名を出力しない。** 「考えられる状態の整理」も疾患の断定ではなく、医師が確認すべき観点の列挙にとどめる。
2. **緊急度判定は「受診行動の目安」であり、医学的判断ではない**ことを画面上に常時明示する（一般的な受診勧奨・救急相談（#7119等）の案内と同等の位置づけ）。
3. 出力はすべて**医師の確認を経て確定**する（医師確認必須フロー）。
4. 患者向け画面には**AIの出力をそのまま「あなたは〇〇病です」の形で表示しない**。
5. LP・広告・営業資料でも「AI診断」という表現を使わない（「AI問診サポート」「問診整理AI」等に統一）。

> ⚠️ 緊急度判定（トリアージ表示）は該当性のグレーゾーンに最も近い機能。
> リリース前に**薬機法に明るい弁護士または PMDA の該当性相談で確認する**ことを必須タスクとする（ユーザー側対応）。

### 2.2 医師法17条（医業）との関係
診断・治療は医行為であり医師にしかできない。AIが単独で診断・処方を行えば無資格医業の幇助になり得る。
→ 本システムの出力は「医師への情報提供」であり、患者への医学的判断の提供ではない、という建付けを全機能で維持する。

### 2.3 オンライン診療指針
厚労省「オンライン診療の適切な実施に関する指針」に準拠する。SaaS側の責務として：

- 医師-患者間のビデオ診療はリアルタイム映像＋音声（本システムはURL発行・予約管理を担い、診療実施の適否判断は医師）
- 本人確認（患者・医師双方）の手段を提供
- 通信の暗号化・なりすまし防止
- 診療計画・診療録は医療機関側の責務だが、システムはその記録を支援

### 2.4 個人情報保護法（要配慮個人情報）
症状・病歴・服薬・妊娠有無はすべて**要配慮個人情報**。

- 取得時に**明示的な本人同意**（同意画面・同意ログの保存）
- オプトアウトによる第三者提供は不可
- 外部AI API（OpenAI/Anthropic）への送信は「委託」構成とし、**学習利用しない契約（DPA / zero data retention オプション）**を締結。外国にある第三者への提供に該当するため、**同意文言に外国送信を明記**
- 削除申請（消去請求）対応フローを実装

### 2.5 3省2ガイドライン
医療情報を取り扱うシステム事業者として：

- 厚労省「医療情報システムの安全管理に関するガイドライン（第6.0版）」
- 経産省・総務省「医療情報を取り扱う情報システム・サービスの提供事業者における安全管理ガイドライン」

に沿った安全管理措置（アクセス制御・監査ログ・暗号化・バックアップ・リスク分担の明確化）を非機能要件に反映する（§5）。
医療機関との契約書で**責任分界点**（診療行為＝医療機関、システム安全管理＝SaaS）を明記する。

### 2.6 その他
- **医療広告ガイドライン**：テナントのクリニック紹介ページを持つ場合、広告規制（体験談・ビフォーアフター等）に留意
- **特定商取引法・資金決済**：SaaS課金部分（本フェーズでは対象外、将来Stripe等）
- **未成年**：小児科は保護者の同意・代理入力を前提とするUI

## 3. 機能要件

### FR-1 患者問診フォーム
| ID | 要件 | 優先度 |
|---|---|---|
| FR-1-1 | 症状（主訴）を選択式＋自由入力で登録できる | MVP |
| FR-1-2 | 発症時期（いつから）を選択できる | MVP |
| FR-1-3 | 痛みの程度（0-10 NRSスケール、顔アイコン併用） | MVP |
| FR-1-4 | 体温を入力できる（未計測も選択可） | MVP |
| FR-1-5 | 既往歴・服薬中の薬・アレルギーを登録できる（2回目以降は前回値を引き継ぎ） | MVP |
| FR-1-6 | 妊娠有無・授乳有無（性別・年齢により表示制御） | MVP |
| FR-1-7 | 生活習慣（喫煙・飲酒・睡眠） | MVP |
| FR-1-8 | 画像アップロード（皮膚症状等、最大5枚、EXIF除去） | MVP |
| FR-1-9 | 自由入力欄（伝えたいこと） | MVP |
| FR-1-10 | 代理入力モード（保護者・家族が入力、続柄を記録） | Phase 2以降 |
| FR-1-11 | 途中保存・再開 | MVP |
| FR-1-12 | 診療科別テンプレート（内科/皮膚科/小児科/メンタル） | 皮膚科・内科のみMVP |

### FR-2 AI追加質問
- 患者の回答をAIが解析し、不足情報を**1問ずつ・最大8問まで**追加質問する（高齢者配慮：一度に複数聞かない）
- 質問は選択式を優先し、自由入力は補助とする
- 危険症状ワード（胸痛・呼吸困難・意識障害・自殺念慮 等）を検知した場合、追加質問を打ち切り緊急案内へ遷移
- AIの全質問・全回答をログ保存（ai_questions / audit_logs）
- AIが応答不能・タイムアウトの場合も問診は完了できる（AIはあくまで補助。フォールバック：定型追加質問）

### FR-3 緊急度判定
4分類で表示する。**表現は不安を煽らず、かつ危険を過小評価させない**こと。

| 区分 | 患者向け表示 | 挙動 |
|---|---|---|
| L1 救急受診推奨 | 「すぐに救急要請（119）または救急外来の受診をご検討ください」 | 問診を中断し全画面アラート。#7119案内。医師・クリニックに即時通知 |
| L2 早急な診療推奨 | 「本日中の受診をおすすめします」 | 当日枠を優先表示 |
| L3 通常診療で相談 | 「通常の診療でご相談ください」 | 通常予約導線 |
| L4 経過観察候補 | 「まずは様子を見ることも考えられます（ご心配なら受診を）」 | セルフケア情報は出さない（医療機器該当性リスク）。受診導線は常に表示 |

- **判定は保守側に倒す**（迷ったら上位区分）。L4は最も慎重に扱う
- 判定根拠（どの回答が効いたか）を医師向けに保存
- 画面に常時「この表示は受診の目安であり、診断ではありません。最終的な判断は医師が行います」を表示

### FR-4 医師向けSOAP要約
- S（主観的情報）：患者の訴えの要約
- O（客観的情報）：体温・入力値・画像有無等の事実
- A（考えられる状態の整理）：**鑑別の断定ではなく「医師が確認すべき観点」の列挙**。確度表現は使わない
- P（医師確認事項）：追加で確認すべき問診項目・見落とし注意点
- 医師は要約を編集でき、編集履歴を保存。**「AI生成・未確認」バッジを医師確認まで表示**

### FR-5 診療科目提案
- 内科・皮膚科・耳鼻科・小児科・婦人科・精神科・整形外科・眼科から**候補を最大2つ**提示
- テナントが標榜していない科は「他院受診のご案内」文言に切替
- 断定表現禁止（「〇〇科が候補です」まで）

### FR-6 オンライン診療予約
- 診療枠管理（クリニック管理者が設定）／医師選択／枠選択
- ビデオ診療URL発行（MVP：Daily.co等の外部API。WebRTC自前実装はしない）
- 予約確認メール（Resend等）／リマインドSMS（Twilio）
- キャンセル・変更（前日まで）

### FR-7 医師管理画面（ダッシュボード）
- 本日の予約一覧・患者一覧・緊急度フィルター（L1/L2を最上部に固定）
- 問診レポート詳細（SOAP・AI質問履歴・画像・過去問診）
- 診療メモ・処方メモ（**フリーテキスト。システムは処方判断に関与しない**）
- PDF出力（問診レポート）
- 医師確認ステータス管理（未確認→確認済み）

### FR-8 患者マイページ
- 問診履歴／診療予約の確認・変更／医師コメント閲覧／処方情報（医師が記載したメモの閲覧）／次回診療予定

### FR-9 AI安全設計（横断要件）
| ID | 要件 |
|---|---|
| SAFE-1 | 全AIプロンプトに診断断定禁止・処方指示禁止の制約を埋め込み、出力を後段バリデーション（禁止パターン検知）する二重ガード |
| SAFE-2 | 危険症状検知は**AI判定＋ルールベース（キーワード・バイタル閾値）の並走**。どちらかが検知したらL1扱い（AI単独に依存しない） |
| SAFE-3 | 自殺念慮・自傷他害の検知時は専用フロー（いのちの電話等の相談先提示＋クリニック即時通知） |
| SAFE-4 | AIの全入出力（プロンプト・応答・モデル・バージョン）を ai_logs に保存、監査可能にする |
| SAFE-5 | 医師確認必須フロー：AI出力は医師の確認操作があるまで「未確定」状態 |
| SAFE-6 | 自由入力欄のプロンプトインジェクション対策（入力のサニタイズ・システムプロンプト分離・出力検証） |
| SAFE-7 | AI障害時のグレースフルデグラデーション（問診自体は完了可能） |

### FR-10 セキュリティ・管理（横断要件）
- ロールベース権限（patient / doctor / clinic_admin / super_admin）＋RLSによるテナント分離
- 医療従事者アカウントは二段階認証必須
- 監査ログ（誰が・いつ・どの患者情報に・何をしたか）
- 保存時暗号化・通信TLS・画像の署名付きURL（短期限）
- バックアップ（日次）・データ削除申請対応
- 個人情報の保持期間ポリシー（テナント設定可能、法定保存に配慮）

## 4. 非機能要件（抜粋）

| 分類 | 要件 |
|---|---|
| 性能 | 問診画面初期表示 2秒以内／AI追加質問の応答 8秒以内（超過時はローディング＋フォールバック） |
| 可用性 | 稼働率99.5%目標。AI APIダウン時も問診・予約は継続可能 |
| ユーザビリティ | スマホ最優先・文字16px以上・WCAG 2.1 AA準拠目標・1画面1〜2質問・所要5〜7分 |
| 拡張性 | マルチテナント（clinic単位）。診療科テンプレートの追加が設定で可能 |
| 監査性 | すべての医療情報アクセス・AI入出力・同意操作をログ保存（改ざん防止：追記専用） |
| データ | 東京リージョン（Supabase ap-northeast-1）。AI APIへの送信は必要最小限の項目に絞る（氏名・連絡先は送らない） |

## 5. リスク分析（リスク登録簿）

| # | リスク | 影響 | 発生可能性 | 対策 |
|---|---|---|---|---|
| R1 | **緊急度の過小評価（false negative）で重篤疾患を見落とす** | 致命的 | 中 | SAFE-2の二重判定／保守側判定／L4でも受診導線常設／免責でなく設計で防ぐ＋規約で最終判断は医師と明記 |
| R2 | 薬機法上、医療機器該当と判断される | 事業停止級 | 中 | §2.1の設計縛り／リリース前に専門家・PMDA該当性確認（**ユーザー側タスク**） |
| R3 | AIのハルシネーション（存在しない症状・誤要約） | 大 | 中 | SOAPに原文参照を併記／医師確認必須／出力バリデーション |
| R4 | 要配慮個人情報の漏えい | 事業停止級 | 低 | RLS・暗号化・監査ログ・最小権限・ペネトレーションテスト（Phase 12） |
| R5 | 外部AI APIへの医療情報送信が同意・契約面で不備 | 大 | 中 | DPA締結・学習不使用・同意文言に外国送信明記・送信項目の最小化 |
| R6 | プロンプトインジェクション（自由入力経由でAI出力を操作） | 中 | 中 | SAFE-6 |
| R7 | 過剰トリアージ（over-triage）で救急・当日枠が逼迫 | 中 | 中 | 判定閾値のチューニング／医師フィードバックループ（判定と医師評価の突合） |
| R8 | 高齢者が入力を完了できない | 中 | 高 | UI方針の徹底・代理入力・途中保存・電話案内の併記 |
| R9 | メンタル領域で自殺念慮への不適切応答 | 致命的 | 低 | SAFE-3専用フロー。AIに励まし・助言をさせない（相談先提示に限定） |
| R10 | ビデオ診療の通信品質・接続不良 | 中 | 中 | 外部API利用（自前WebRTC回避）／接続テスト画面／電話フォールバック案内 |
| R11 | テナント間データ混在（マルチテナントバグ） | 大 | 低 | RLSをDB層で強制＋テストで検証（Phase 13） |
| R12 | 医師がAI要約を鵜呑みにする（automation bias） | 大 | 中 | 「AI生成・未確認」バッジ／原文併記／確認操作の必須化 |

## 6. MVP範囲

### 6.1 MVPに含む（Phase 5〜11で実装）
1. 認証（患者・医師・クリニック管理者、医療従事者2FA）
2. 患者問診フォーム（内科・皮膚科テンプレート、画像アップロード含む）
3. AI追加質問（最大8問・フォールバック付き）
4. 緊急度判定（4分類・二重ガード・救急アラート）
5. 医師向けSOAP要約＋医師確認フロー
6. 診療科目提案
7. 医師ダッシュボード（一覧・緊急度フィルター・問診詳細・診療メモ）
8. 予約（枠管理・外部ビデオAPI連携・確認メール）
9. 患者マイページ（履歴・予約確認）
10. PDF出力（問診レポート）
11. 監査ログ・同意取得・RLSテナント分離

### 6.2 MVPに含まない（Post-MVP）
- SMS通知（MVPはメールのみ。Twilio導入はPhase 10拡張）
- 小児科・メンタル専用テンプレート（メンタルはSAFE-3の設計が重いため第2弾。ただし危険ワード検知自体はMVPに含む）
- 代理入力モード／多言語対応／決済（SaaS課金）／電子カルテ連携（ORCA等）／処方箋連携／企業健康相談窓口向け機能／チャット（医師-患者の非同期メッセージ）／管理者向け分析ダッシュボード

### 6.3 MVP成功基準（例）
- 問診完了率 80%以上（開始→送信）
- 医師のSOAP修正率を計測できる状態（AI品質のKPI化）
- L1検知の見逃しゼロ（テストシナリオ集で検証）

| 指摘 | 対応 |
|---|---|
| 「経過観察候補」の表示は受診抑制につながり、見落とし時の責任問題が最も重い | L4でも受診導線を常設し、セルフケア助言は出さない仕様に変更（FR-3） |
| AI単独の緊急度判定は、AI障害・ハルシネーションで危険 | ルールベース並走（SAFE-2）とフォールバックを要件に追加 |
| 氏名・連絡先までAI APIに送る必然性がない | 送信項目の最小化を非機能要件に明記（§4） |
| メンタルクリニック対応をMVPに含めると自殺リスク対応の設計負債になる | メンタル専用テンプレートはPost-MVPに後ろ倒し。ただし危険ワード検知は全科共通でMVPに残す |
| WebRTC自前実装は工数・品質リスクが大きい | 外部ビデオAPI（Daily.co等）採用に決定 |
| 「診断診断サポート」的な命名・文言は薬機法リスク | プロダクト文言を「問診サポート」系に統一する方針を§2.1に明記 |
| 処方メモ機能がAIと絡むと処方指示に見える | 処方メモは医師のフリーテキストのみ、AI非関与と明記（FR-7） |
