# Phase 6：問診フォーム 実装メモ（実装前設計＋自己レビュー）

- 作成日：2026-07-02

## 1. 実装スコープ
1. 問診データ層：`QuestionnaireRepo`（interface＋Prisma実装＋デモ実装）。全メソッド AuthContext 必須（tenant guard）
2. API：作成／取得／ステップ保存（PATCH）／送信（submit）／画像登録
3. 患者UI：テンプレート選択（内科系/皮膚科系）→ 基本7ステップウィザード → 送信 → 追加質問スタブ（Phase 7）
4. **ルールベース危険判定エンジン**（SAFE-2 の片翼）：全ステップ保存時に同期実行（<10ms）。L1級ヒットで S-07E 緊急アラート画面へ
5. 自殺念慮の検知（SAFE-3）：専用の相談先表示バリアント
6. マイページ：問診の開始・再開・履歴一覧

## 2. 設計判断
| 論点 | 判断 | 理由 |
|---|---|---|
| ウィザードの状態管理 | 1ステップ=1ルート（`/questionnaire/[id]/step/[n]`）。各ページはサーバーで問診を取得し、そのステップ分だけのクライアントフォームを描画 | 巨大なクライアント状態を持たない。途中保存＝ステップ保存そのものになり、再開はcurrentStepへ飛ぶだけ |
| 緊急リダイレクト | PATCHレスポンスの `emergency` を見てクライアントが遷移。**既にflag済みの問診では再遷移しない**（「続ける」を選んだ患者を毎ステップ遮らない） | Phase 2 S-07E の設計。通知・ログは初回検知時のみ |
| L2ヒットの扱い | 問診は中断しない（S-07EはL1級と自殺リスクのみ）。L2ヒットはPhase 8のルール判定で再計算されるため保存不要（ルールは決定的） | 過剰な中断は離脱を生む |
| 体温「測っていない」 | bodyTemp=null を「未計測」として扱う（専用カラムを増やさない） | スキーマ簡素化 |
| 既往歴等の入力 | 自由記述1欄ずつ＋「特になし」ボタン。前回値（patients.baseline*）を初期値に表示し、送信時に書き戻す | Phase 1 FR-1-5。チップUI等の作り込みはPost-MVP |
| 画像 | クライアントでcanvas再エンコード（JPEG化・長辺1600px）→ base64でAPIへ。デモ：メモリ保存／本番：service roleでStorageへ | canvas再エンコードでEXIF（位置情報）は自然に除去される。署名URL方式（PHASE4）より単純なMVP実装に変更 |

## 3. 自己レビュー（実装前）
| 指摘 | 対応 |
|---|---|
| クライアント再エンコードだけではEXIF除去をサーバーが保証できない（改造クライアントで生データを送れる） | 既知の残課題として Phase 12 でサーバー側再処理（sharp）を追加予定。本フェーズでは images API が JPEG/PNG/WebP 以外と3MB超を拒否 |
| PHASE4 で決めた署名URL方式から base64 直POST に変えた | 設計書との差分として本メモに明記。Vercel のボディ上限（4.5MB）内に収まるようクライアントで圧縮 |
| 危険キーワードが自由入力の否定文（「息苦しさはない」）にも反応する | MVPは**過剰検知を許容**（Phase 1 R1：見落としの方が致命的）。S-07Eに「症状が落ち着いている・続ける」導線があるため実害は小さい。精度改善はAI側（Phase 7の⑥）が担う |
| ステップ保存のたびに全文キーワード走査するコスト | 対象テキストは数百文字・キーワード数十個の includes 走査で<1ms。問題なし |
| 妊娠質問を性別で出し分ける際、患者マスタのsexを参照する必要がある | GET問診レスポンスに `patientSex`・`patientAge` を含める（PII最小限：氏名は返さない） |
| 皮膚科テンプレートで痛みNRSや体温が冗長 | MVPでは全ステップ共通とし、皮膚科は主訴カテゴリ初期値＋画像案内文言のみ変える（テンプレ分岐の作り込みはPost-MVP） |
